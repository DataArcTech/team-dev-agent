# 完整开发工作流

## 阶段 1：架构设计（先问再动）

在写一行代码前，必须确认：

**功能层面**
- 核心功能是什么？MVP 边界在哪里？
- 用户是谁，操作路径是什么？
- 有哪些角色/权限需要隔离？

**技术层面**
- 平台：Web / 移动端 / 小程序？
- 后端框架：FastAPI / Express / Django / Flask？（基于需求选，不固定）
- 前端框架：Next.js / React / Vue / 微信小程序？
- DB：SQLite（原型）→ PostgreSQL（生产）/ MySQL / MongoDB？
- 需要哪些外部 API / 第三方服务？

**工程层面**
- 单体 vs 微服务？（B 端产品原型期统一用单体）
- 需要管理后台吗？
- 部署目标：本地 / 服务器 / 云？

**输出**：写一份 `ARCHITECTURE.md`，包含：
1. 功能清单（按优先级排序）
2. 技术栈决定 + 简单理由
3. DB 表结构草图
4. 分期开发计划（P1/P2/P3）

## 阶段 2：Claude Code 实现

### 2.1 后端先行

```
DB schema → CRUD API → 业务逻辑 → Auth
```

**验收标准**：
- 所有端点可以用 curl / httpie 测通
- DB 有真实种子数据（不是 mock，而是接近真实场景的 seed）
- 日志清晰，错误处理完善

**给 Claude Code 的 prompt 模板**：

```
## 任务：实现后端 API

项目路径：[PATH]
技术栈：[FastAPI/Express/etc]
DB：[SQLite/PostgreSQL/etc]

## 需要实现
1. DB 表：[列举核心表]
2. API 端点：[列举]
3. Auth：[JWT / Session / 无]

## 数据要求
- 创建 seeds.py / seed.sql，写入真实场景的测试数据（不是 "test1", "test2" 这种）
- 所有计算逻辑在后端完成，前端只做展示

## 铁律
- 禁止 mock 数据 / 假数据 / hardcode 数值
- 如果某个需求无法实现，明确说明原因，不要用假数据绕过
- 完成后跑测试确认全部通过
```

### 2.2 前端骨架（跑通为主，不求漂亮）

**顺序**：路由结构 → 页面骨架 → API 调用 → 数据展示

**验收标准**：
- 每个页面能显示真实数据（从 DB 来的）
- 表单提交能写入 DB
- 页面间跳转正常

### 2.3 前后端联调

重点检查：
- API 请求 URL 是相对路径还是写死了 localhost？
- WebSocket 连接地址是否动态读取？
- 环境变量（API key、DB 路径）是否通过 `.env` 传入，不硬编码？

## 阶段 3：Codex 前端视觉优化

骨架跑通后，交给 Codex 做视觉升级。**不改业务逻辑，只改视觉层**。

用法：见 `frontend-prompts.md` 里的标准 prompt，按需选用组合。

**典型优化序列**：
1. 全局设计风格（选一种：玻璃拟态 / 深色科技感 / 极简白）
2. 布局优化（间距、对齐、响应式）
3. 交互微动效（hover、toast、loading）
4. 去 emoji → icon（必做）
5. 表单体验优化（实时校验、错误提示）

## 阶段 4：质量关卡（2 轮 review）

### 第 1 轮：Bug Fix

```
给 Claude Code 的 prompt：

对 [项目路径] 进行全面 bug review：
1. 运行所有测试，列出失败项并修复
2. 检查所有 API 错误处理（空值、类型错误、权限边界）
3. 检查前端异常处理（网络失败、空数据、加载态）
4. 修复发现的所有 bug

修复完成后：
- 再跑一遍测试，确认全部通过
- 列出修复了哪些 bug
```

### 第 2 轮：去冗余 + 去硬编码

```
给 Claude Code 的 prompt：

对 [项目路径] 进行代码质量 review：

1. 去硬编码
   - 扫描所有 hardcoded URL、IP、端口，移到 .env / config
   - 扫描 mock 数据、test 数据混在业务逻辑里的情况
   - 扫描硬编码的用户 ID、token、密码

2. 去冗余
   - 删除未使用的 import、变量、函数
   - 合并重复逻辑（DRY）
   - 删除 console.log / print 调试语句

3. 可维护性
   - 复杂函数拆分（单函数不超过 50 行）
   - 关键逻辑加注释
   - 常量提取到 constants 文件

完成后提交 commit，commit message 格式：
refactor: remove hardcodes + dead code + improve maintainability
```

## 阶段 5：数据打通测试

**这是最重要的阶段**，专门验证"不是 demo"。

### 5.1 前后端连通性

```bash
# 给 Claude Code 的 prompt：
对 [项目路径] 进行前后端数据打通测试：

1. 列出所有 API 端点
2. 对每个端点：
   - 确认前端确实在调用（不是写死的假数据）
   - 确认后端返回真实 DB 数据
   - 确认前端正确展示了返回数据
3. 修复所有发现的数据断点
```

### 5.2 DB 真实数据验证

- 在 DB 里改一条数据，刷新页面，确认展示变了
- 通过前端表单新增一条，确认 DB 里出现了
- 删除一条，确认 DB 里消失了

### 5.3 功能间联动

- A 功能的操作是否影响了 B 功能的数据显示？
- 权限变更后，受影响的页面是否正确更新？

### 5.4 业务系统 ↔ 管理后台打通

- 管理后台改配置/数据，业务系统立即生效？
- 业务系统的操作，管理后台的日志/统计是否更新？

## 阶段 6：边界 & 交互测试

**每轮一个测试重点**，给 Claude Code 明确的测试清单。

### 边界测试清单模板

```
第 1 轮 — 输入边界
- 空字符串、null、undefined 输入
- 超长字符串（10000 字符）
- 特殊字符（<script>、SQL 注入、Unicode）
- 负数、超大数、小数

第 2 轮 — 权限边界
- 未登录访问受保护页面
- 低权限用户访问高权限接口
- 跨用户访问他人数据

第 3 轮 — 并发 & 异常
- 快速重复提交表单
- 网络断开时的操作
- 后端返回 500 时前端表现
- 超时处理

第 4 轮 — 数据边界
- 空列表（0 条数据）
- 大量数据（1000+ 条）
- 分页边界（第一页/最后一页/超过总页数）
```

### Web 端交互测试（Chrome DevTools）

用 browser 工具 + chrome-devtools skill 实现自动化点击测试：

```
给 Claude Code 的 prompt：

使用 Chrome DevTools MCP 对 [URL] 进行交互测试：

1. 导航到主页，截图确认加载正常
2. 点击 [按钮/链接]，等待响应，截图
3. 填写表单：[字段: 值]，提交，确认成功提示出现
4. 检查 Network 面板，确认 API 调用返回 200（不是 mock）
5. 检查 Console，确认没有 JS 错误

发现问题自动修复，修复后重新测试。
```

## 阶段 7：迭代 review

每次迭代，Claude Code 从一个角度 review，不要同时改太多：

```
角度 1 — 可维护性
"找出耦合度高、难以修改的模块，提出重构方案并实现"

角度 2 — 可扩展性  
"找出如果要加 [新功能X] 需要改哪些地方，是否合理？给出改进方案"

角度 3 — 性能
"找出明显的性能瓶颈：N+1 查询、大量 DOM 操作、不必要的重渲染"

角度 4 — 安全
"扫描：SQL 注入、XSS、敏感数据暴露、权限绕过"
```
